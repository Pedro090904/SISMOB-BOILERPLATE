6. Proteção contra SQL Injection

flowchart TD A["FilterLinhaDto\n{codigo?, descricao?}"] --> B["LinhaRepository\nfindByFilters"] B --> C["SELECT ... WHERE 1=1"] C --> D{codigo\nfornecido?} D -- Sim --> E["sql += AND cd_linha = $1\nparams.push(codigo)"] D -- Não --> F[Ignora] E --> G{descricao\nfornecida?} F --> G G -- Sim --> H["sql += AND tx_linha LIKE $2\nparams.push('%desc%')"] G -- Não --> I[Ignora] H --> J["dataSource.query(sql, params)\n✅ Driver substitui $1,$2 com segurança"] I --> J J --> K[(PostgreSQL)]
⚠️ NUNCA faça sql += "AND cd_linha = '" + codigo + "'". Isso permite SQL Injection! Use sempre parâmetros $1, $2...

16. Desafios de Desenvolvimento (Para Estagiários)

💡 Cloneu o repositório? Teste seus conhecimentos resolvendo os seguintes desafios propostos para consolidar o entendimento desta arquitetura:
  1. CRUD Completo de Linha: Implementar as rotas que faltam (POST, PUT, DELETE e GET por ID) para a entidade linha. Obrigatório: Utilizar restritamente todas as pastas da arquitetura (Guards para segurança, DTOs para validação, Utils, Exceptions personalizadas, Filters, Controller, Service e Repository).
  2. Novas Entidades, CRUDs e Dados Espaciais: Criar as entidades tab_operadora, tab_operadora_linha e tab_itinerario. Desenvolver o CRUD completo para elas, sendo que tab_itinerario deverá conter dados espaciais (geometrias/geography) para lidar com a rota cartográfica e desafiar o manuseio desse tipo de dado no NestJS/TypeORM. É obrigatório passar por todas as pastas da arquitetura para esses novos recursos (Module, Controller, Service, Repository, DTO, Entity), além de aplicar a camada de segurança (Guards) e Utils.
  3. Relacionamentos e Seed: Relacionar as entidades para ser possível buscar a operadora de uma determinada linha (via Operadora_Linha) e buscar o traçado cartográfico da linha (via relação Linha com Itinerario). O estagiário também deverá criar as SEEDS e a automação de inserção inicial no banco para essas novas tabelas, inclusive inserindo o dado espacial (semelhante ao arquivo tab_linha.sql).
  4. Containerização Docker: Subir o banco de dados e a aplicação utilizando Docker e o arquivo docker-compose.yml fornecido, garantindo que o backend funcione corretamente dentro do container e se conecte com o banco.
  5. Build e Push para o GitLab: Fazer o build do projeto SISMOB-BOILERPLATE gerando uma imagem Docker utilizando o Dockerfile fornecido, e realizar o envio (push) dessa imagem para o repositório de containers do GitLab.

17. Comandos Úteis (Cheat Sheet)

Guia rápido com todos os comandos necessários no dia a dia, desde baixar o código até buildar a imagem Docker:

🦊 Git & Versionamento

# Clonar o repositório
git clone <url-do-repositorio>

# Enviar alterações
git add .
git commit -m "feat: adicionando CRUD de operadora"
git push origin main

# Criar e enviar Tag (dispara o Pipeline no GitLab!)
git tag v1.0.1
git push --tags

🔷 Node.js & NestJS

# Instalar dependências iniciais
npm install

# Rodar aplicação local com auto-reload
npm run start:dev

# Compilar para produção (gera pasta /dist)
npm run build

# Atalhos práticos do Nest CLI (cria os arquivos base)
npx nest g module operadora
npx nest g controller operadora
npx nest g service operadora

🐳 Docker Compose (Ambiente Local)

# Subir banco de dados e pgAdmin
docker-compose up -d

# Visualizar logs do banco em tempo real
docker-compose logs -f sismob-db

# Parar os containers
docker-compose down

# Parar containers E APAGAR DADOS DO BANCO (reset)
docker-compose down -v

📦 Docker Image (Build e Push)

# Fazer o Build da imagem local (Atenção ao "." no final!)
docker build -t sismob-app .

# Autenticar no Registry (ex: GitLab)
docker login registry.gitlab.com -u <usuario>

# Nomear (Taggear) a imagem para o repositório correto
docker tag sismob-app registry.gitlab.com/projeto/sismob-app:latest

# Enviar (Push) a imagem para o repositório
docker push registry.gitlab.com/projeto/sismob-app:latest
